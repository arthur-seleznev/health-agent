# CLAUDE.md — health-agent

Контекст проекта для Claude Code. Читай целиком перед работой.

## Что это

Персональный агент для осознанного питания: на вход — запрос пользователя
(«сытный белковый обед»), на выход — структурированное блюдо с ингредиентами,
а в перспективе — расчёт БЖУ по реальной базе продуктов, список покупок,
трекинг и напоминания (врач, анализы, вес).

背景: пользователь начинает следить за питанием. Точность БЖУ важна — это
агент-калькулятор, поэтому ключевой принцип: **модель оркестрирует, а не считает.**
LLM называет продукты и количества, числа БЖУ берутся из выверенной базы, не из
головы модели.

## Стек

- **NestJS 11** (TypeScript, ESM, `module: nodenext`) — каркас, DI.
- **LangGraph.js 1.x** (`@langchain/langgraph`) — оркестрация графа агента.
- **@langchain/openai** — клиент к LLM через OpenAI-совместимый endpoint.
- **Zod 4** — сквозная валидация: стейт графа, DTO входа, контракты нод.
- **nestjs-zod** — Zod-DTO + `ZodValidationPipe`.
- **pnpm** (не npm!) — единый пакетный менеджер, он же управляет Node (LTS 24).
- **LLM:** локальная **MiniMax-M2.7** через **SGLang**, OpenAI-совместимый endpoint
  на ai2-сервере (`LLM_BASE_URL` в `.env`, доступ через WireGuard).
- Линтер/форматтер: ESLint + Prettier. `trailingComma: none` (по предпочтению владельца).

## Архитектурные решения и ГРАНИЦЫ (не нарушать)

Эти решения приняты осознанно владельцем в ходе проектирования. Не «упрощай» их
обратно — за каждым стоит причина.

1. **LlmService — инкапсулированная обёртка.** Клиент `ChatOpenAI` ПРИВАТНЫЙ
   (`private readonly chat`). Наружу только узкие методы `invoke(messages)` и
   `invokeStructured(messages, schema)`. НЕ выставлять `chat` публично, НЕ
   наследовать LlmService от ChatOpenAI (композиция, не наследование). Причина:
   обёртка — место для будущего логирования (Langfuse), retry, выбора модели.

2. **Ноды графа — это @Injectable провайдеры**, а не функции-замыкания. Метод-нода
   объявляется как СТРЕЛОЧНОЕ СВОЙСТВО (`run = async (state) => {...}`), НЕ обычный
   метод. Причина: LangGraph вызывает ноду оторванно от объекта — обычный метод
   потеряет `this`, стрелочное свойство захватывает контекст навсегда. Зависимости
   (LlmService) инжектятся в ноду нативным DI через конструктор.

3. **Граф собирается в MenuService.onModuleInit()**, не в конструкторе, не отдельным
   провайдером с токеном. MenuService — единственный фасад над графом (топология
   в приватном `buildGraph()`). Не плодить лишние слои-обёртки над компилированным
   графом. Тип графа: `ReturnType<typeof this.buildGraph>`.

4. **DTO и State РАЗДЕЛЕНЫ.** DTO — контракт HTTP-границы (`dto/`). State — контракт
   рантайма графа (`menu.state.ts`). НЕ объединять, даже если похожи: меняются по
   разным причинам, используют разные обёртки Zod (nestjs-zod vs langgraph StateSchema).
   Маппинг между ними — работа фасада (`suggestMeal(dto.request)`).

5. **Контракты нод (схемы structured output) живут в `nodes/contracts/`**, отдельным
   файлом (не внутри файла ноды). Причина: и нода, и стейт импортируют схему — отдельный
   файл разрывает циклическую зависимость state↔node.

6. **MealSchema НЕ содержит калорий/БЖУ** — только `product`, `qty`, `unit`, `steps`.
   Числа питательности считаются отдельной нодой из базы. `unit` — это ENUM
   (`g/ml/piece/tbsp/tsp/pinch`), НЕ свободная строка. Причина: закрытое множество
   единиц нужно для надёжного матчинга с базой продуктов.

7. **Тонкие контроллеры не покрываются юнит-тестами** (тавтология). CLI-заглушки
   `*.spec.ts` удалены. Контракты API — через e2e (`test/`). Юнит-тесты пишутся для
   логики нод/сервисов, когда она появляется (напр. расчёт БЖУ).

8. **Версии и API не выдумывать.** LangGraph.js, Zod, nestjs-zod быстро меняются.
   Если есть сомнение в актуальности API — проверь доку/исходники, не пиши по памяти.
   Это прямое требование владельца.

## Текущее состояние (что работает)

- `POST /menu/suggest` с телом `{"request": "..."}` → возвращает структурированный
  объект Meal (name, ingredients[], steps[], cookTimeMinutes). Валидация входа через Zod.
- Полная цепочка Nest → LangGraph (1 нода `menu_suggestion`) → LlmService →
  structured output (`withStructuredOutput`) → MiniMax/SGLang. **Подтверждено: SGLang
  отдаёт валидный structured output из коробки, без extra_body/method.**
- Граф пока из одной ноды: START → menu_suggestion → END.

## Следующая большая задача: база продуктов + нода nutrition_check

### База — Postgres + pgvector (НЕ ClickHouse)

ClickHouse отвергнут осознанно: данных мало (~7793 продукта), нагрузка — точечный
поиск по имени + семантический матчинг, а не OLAP-агрегации. Postgres+pgvector
закрывает и реляционное хранение, и векторный поиск названий. ClickHouse в проекте
остаётся для аналитики/дашбордов (Superset, Langfuse-бэкенд), не для этой витрины.

### Источник данных: USDA SR Legacy (public domain)

CSV-дамп USDA уже скачан (лежит в uploads / надо положить в проект). Ключевое:
- `food.csv` — 7793 продукта (fdc_id, description, food_category_id).
- `food_nutrient.csv` — 644125 строк связки продукт↔нутриент (fdc_id, nutrient_id, amount).
- `nutrient.csv` — справочник нутриентов.
- `food_category.csv` — 28 категорий.
- Значения нутриентов — НА 100 Г съедобной части.
- Калории (nutrient_id=1008) есть у ВСЕХ 7793 продуктов (100% покрытие).

Нужные nutrient_id для фильтра импорта:
- Макро: 1008 kcal, 1003 protein, 1004 fat, 1005 carb, 1079 fiber, 2000 sugar.
- Микро (хранить, но агент читает позже): 1093 sodium, 1092 potassium, 1087 calcium,
  1090 magnesium, 1089 iron, 1095 zinc, 1106 vit_a, 1162 vit_c, 1114 vit_d,
  1175 vit_b6, 1178 vit_b12, 1177 folate.

### Схема витрины (денормализованная, спроектирована, НЕ финализирована)

Плоская таблица `products`: id (fdc_id), name, category, + колонки нутриентов
(на 100 г, тип `real`), `source` (провенанс: 'usda'/'model_estimate'), `embedding`
vector(N), таймстемпы. Индексы: HNSW на embedding (vector_cosine_ops), GIN trgm на name.

ОТКРЫТЫЕ ВОПРОСЫ (решить с владельцем, НЕ решать самому):
- PK: натуральный fdc_id или суррогатный + external_id? (влияет на не-USDA источники)
- Embedding-модель и размерность vector(N)? Нужна МУЛЬТИЯЗЫЧНАЯ (меню от модели
  по-русски, USDA по-английски — кросс-язычный матчинг). Размерность фиксировать
  ДО генерации векторов.
- Категории: текстом или id+справочник?

### Нода nutrition_check (план)

Берёт ingredients[] из стейта → для каждого матчит продукт в базе (двухступенчато:
сначала pg_trgm точный/триграммный, потом pgvector семантический) → считает БЖУ
по qty+unit и значениям на 100г → кладёт в стейт. Условное ребро (`addConditionalEdges`):
если продукт не найден — ветка обработки (на старте: пометить «нет данных», НЕ
веб-поиск). Предохранитель от зацикливания (retryCount в стейте через ReducedValue).

ВАЖНО про матчинг (проблема №1 всего агента): продукт из меню по-русски, USDA по-
английски, формулировки разные. Решается pgvector + мультиязычный эмбеддер. На старте
можно проще, но матчинг — главный риск, заложить аккуратно.

### Русский слой (отложено)

USDA — американская номенклатура, русской специфики (творог по ГОСТ и т.п.) нет.
Легальной электронной версии справочника Тутельяна НЕ существует (только бумага).
Пиратские копии (studfile, dony и т.п.) НЕ использовать. Решение на старте: USDA +
для русских продуктов model_estimate с пометкой source. Книгу владелец может купить
бумажную и отсканировать сам — тогда парсить его легальный экземпляр.

## Стиль и тон работы с владельцем

- Владелец — опытный (10+ лет, бэкграунд в банковском treasury + production AI на
  Python/LangGraph), но в TypeScript/Nest начинающий. Объясняй ДЕЛЬТУ от Python, не

- Не срезай архитектурные углы «чтобы работало» — владелец это ловит и не любит.
  Объясняй trade-offs, предлагай варианты с обоснованием, не навязывай.
- Двигаться маленькими шагами с подтверждением, не вываливать всё разом.
- Версии библиотек и API проверять, не выдумывать (см. границу №8).
